# Guided Workflow Imported Plan Adoption Design

## Scope

This document designs future explicit adoption of a read-only imported plan candidate into live Guided draft state.

It does not implement adoption.

It does not implement RunSpec generation.

It does not implement Guided Run.

It does not implement plan-to-RunSpec mapping.

It does not run analysis.

It does not write outputs.

Adoption is a future mutation boundary. Export, open/review, adopt, and run must remain separate stages:

- Export writes the current live draft plan to one explicit JSON file.
- Open/review reads external JSON into separate read-only candidate state.
- Adopt explicitly mutates live Guided draft state from an eligible candidate.
- Run executes analysis from executable configuration, which remains outside the current Guided Workflow stages.

## Existing State Audit

The accepted current state is:

- GuidedRunPlan export exists from the Draft Plan step.
- Draft Plan contains feature/event profile planning, output policy planning, export, readiness summary, draft preview, checklist, and imported plan review.
- Imported plan review can open an exported GuidedRunPlan JSON as a read-only candidate.
- Failed candidate opens clear stale candidate state.
- Candidate review state is separate from live draft state.
- Live readiness, preview, and checklist describe live draft state only.
- Candidate review has its own summary and does not satisfy live readiness.
- No Adopt button exists.
- No RunSpec generation exists.
- No Guided Run exists.

Current candidate state lives in:

- `_guided_imported_plan_candidate`
- `_guided_imported_plan_file_path`
- `_guided_imported_plan_status`
- `_guided_imported_plan_review_run_dir`

Current live Guided draft state lives in separate stores:

- Correction strategy choices: `_guided_strategy_choices`, keyed by completed-run source and ROI.
- Feature/event profiles: `_guided_draft_feature_event_profiles_by_run`, keyed by completed-run source.
- Output policy: `_guided_draft_output_policy_by_run`, keyed by completed-run source.
- Export path editor state: `_guided_export_path_edit` plus `_guided_export_editor_synced_run`.
- Current completed-run source: `_current_run_dir` / `_current_guided_completed_run_dir()`.
- Current visible GUI widgets: volatile controls that do not mutate draft state until explicit Mark or Apply actions.
- Full Control state: separate controls and RunSpec-backed execution paths.

`_build_guided_draft_run_plan()` builds the live draft plan from the live stores only. It must not read candidate state. Future adoption must preserve this separation until the explicit adoption commit point.

## Non-Negotiable App-Level Rules

1. The app is for long-duration photometry analysis, not short-session viewing.
2. Guided Workflow is a planning/review scaffold, not a duplicate Full Control panel.
3. Passive review actions must not run analysis.
4. GuidedRunPlan is planning/provenance, not executable.
5. RunSpec is separate and must not be generated by adoption.
6. Filesystem writes must remain explicit.
7. Full Control must remain separate.
8. No silent fallback, no auto-selection, no auto manifest population.
9. Signal-Only F0 is explicit only, not fallback.
10. OutputPolicy is planning state, not a filesystem operation.
11. Current visible ROI/chunk is not production scope.
12. Evidence chunks are provenance/QC context, not production scope.
13. Live readiness/checklist describe live draft state only.
14. Candidate readiness/review is separate from live readiness/checklist.
15. Opening a plan is review; adopting a plan is mutation; running a plan is execution.

## Adoption Recommendation

Stage 4H2 should remain design-only. A later implementation stage should use explicit, source-matched, replace-only adoption.

Recommended future behavior:

1. User opens candidate plan for read-only review.
2. Candidate review computes adoption eligibility but does not mutate live state.
3. If the candidate passes adoption eligibility checks, a future UI may enable an Adopt button.
4. User clicks Adopt.
5. If live draft state is non-empty, the app shows a confirmation summarizing what will be replaced.
6. On confirmation, the app replaces live draft state for the current completed-run source with the candidate planning state.
7. The app synchronizes Guided widgets only after successful adoption.
8. The app refreshes live readiness, preview, and checklist only after successful adoption.

Adoption must not:

- Change the export path.
- Change the current completed-run source.
- Load or switch to another completed run.
- Change Full Control.
- Generate RunSpec.
- Run analysis.
- Write outputs.
- Run previews or diagnostics.
- Merge candidate state into live state.

Do not implement merge adoption in the first implementation. Merge adoption creates conflict-resolution questions for per-ROI strategies, profile replacement, partial plans, stale evidence, and output policies. Replace-only adoption is easier to reason about and easier to test for no partial mutation.

## Adoption Eligibility Checks

Adoption must be enabled or performed only when all required gates pass. `validate_plan_contract` is necessary but not sufficient for adoption. Source compatibility, ROI compatibility, feature/event compatibility, and OutputPolicy safety are separate adoption gates.

Required gates:

1. Candidate exists.
2. Candidate was produced by successful read-only open/review.
3. Candidate state has not been cleared by a failed open.
4. Candidate file path is still associated with the displayed candidate.
5. Candidate contract validation passes.
6. Candidate schema version is supported.
7. Candidate mode is `completed_run_planning`.
8. Current completed run is loaded.
9. Imported source resolves to the same path as current completed run.
10. Current completed-run source is valid.
11. Imported source is valid, or at minimum matches the current valid source exactly after path resolution.
12. ROI compatibility passes.
13. Missing imported ROIs block adoption.
14. Duplicate imported ROIs block adoption.
15. Unknown ROI status blocks adoption.
16. Stale or unavailable evidence chunks must be warned and preserved as provenance; first implementation may allow them, but they must never trigger reruns.
17. Correction strategies are all valid runnable strategies.
18. Forbidden strategies block adoption: `auto`, `needs_review`, `no_correction`.
19. Unknown strategies block adoption.
20. Signal-Only F0 is allowed only when explicitly present as the candidate strategy.
21. Feature/event profiles pass current validation.
22. Chunk-scoped feature/event profiles block adoption.
23. Unknown feature/event config fields block adoption.
24. Unsupported feature/event values block adoption unless there is a faithful preservation strategy that does not reinterpret them.
25. OutputPolicy is separately revalidated against current source.
26. Unsafe output root blocks adoption.
27. `overwrite=True` blocks adoption initially.
28. `separate_from_source_required=False` blocks adoption.
29. `legacy_outputs_protected=False` blocks adoption.
30. Current read-only candidate review may continue to say adoption unavailable until implementation, but future eligibility must be computed from fresh checks at adoption time.

Eligibility should be recomputed immediately before adoption. Cached candidate review status may be stale if the source changes, ROI inventory changes, files disappear, or the user opens a different candidate.

## Source Matching And Completed-Run Validity

First adoption implementation should be source-matched only.

Rules:

- Normalize imported and current source paths with real/resolved paths before comparison.
- Resolve path spelling differences, relative segments, and symlinks before matching.
- No current completed run loaded blocks adoption.
- Current completed run loaded and matching imported source after resolution can pass the source-match gate.
- Imported source differing from current source blocks adoption.
- Imported source missing blocks adoption unless it resolves to the current loaded source through a still-valid current context, which should be treated cautiously and documented in the eligibility status.
- Imported source existing but not a completed run blocks adoption.
- Raw input mode blocks adoption in this completed-run adoption design.
- Automatic source switching is forbidden.
- Adoption must not load a different completed run.
- Adoption must not change current source.

Completed-run validity should reuse the same conservative source assumptions used by read-only review: successful completed-run evidence plus readable completed-run cache/artifacts when needed for ROI inventory. Source existence alone is not sufficient.

## ROI And Chunk/Evidence Rules

ROI names must be compared against current completed-run ROI inventory. Do not invent current ROI names from imported JSON.

Rules:

- Missing imported ROIs block adoption.
- Duplicate imported ROI entries block adoption.
- Unknown ROI statuses block adoption.
- Extra current ROIs not in the candidate are allowed as partial/incomplete draft state.
- Zero ROI choices should be blocked in the first adoption implementation unless a later design adds an explicit "adopt incomplete empty draft" confirmation. The safer first implementation is to block zero ROI choices.
- Evidence chunks are provenance only.
- Evidence chunks must not become production scope.
- Stale or unavailable evidence chunks must be displayed after adoption if adoption is allowed.
- Adoption must not run preview or diagnostic generation to refresh evidence.
- Evidence chunk availability warnings should remain separate from ROI compatibility.

Recommendation: first implementation should allow extra current ROIs as incomplete draft state, block missing imported ROIs, and block zero ROI choices.

## Correction Strategy Adoption Rules

Correction strategies copy into `_guided_strategy_choices` only after explicit adoption.

Rules:

- Adopted strategies are scoped to the current completed-run source.
- Adopted strategies are ROI-level choices, not chunk-level production scope.
- Evidence chunk remains provenance associated with the strategy choice.
- Signal-Only F0 remains explicit and must be clearly labeled.
- Signal-Only F0 adoption must not run Signal-Only F0 diagnostic.
- `auto`, `needs_review`, and `no_correction` cannot be adopted.
- Unknown strategies cannot be adopted.
- Adoption must not run previews, diagnostics, applied-dF/F routing, manifests, correction execution, or pipeline execution.

Confirm Strategy widget synchronization after adoption:

- If the current selected ROI has an adopted strategy, display that strategy as the current marked choice.
- Evidence labels may show adopted evidence context as provenance.
- Do not infer or auto-select strategies for unconfigured ROIs.
- Do not change the current selected ROI unless the current selection is no longer valid after source/ROI sync. If a change is required, it should follow existing combo refresh behavior and be documented as UI synchronization, not strategy inference.

The replacement conversion should build a complete new choices mapping for the current run in local variables before mutating `_guided_strategy_choices`.

## Feature/Event Profile Adoption Rules

Feature/event profiles copy into `_guided_draft_feature_event_profiles_by_run` only after explicit adoption.

Rules:

- Profiles remain run-level or allowed scope only.
- Chunk scope blocks adoption under the current contract.
- Unknown or invalid config fields block adoption.
- Unsupported values block adoption unless they can be faithfully preserved without reinterpretation and without breaking widget sync.
- Preview/evidence references remain provenance only.
- Adoption must not run feature extraction.
- Adoption must not auto-generate event outputs.
- Feature/event editor widgets synchronize only after successful adoption.
- Widget synchronization must not create additional plan state beyond the adopted state.

Cases:

- No profile in candidate: first implementation may either block adoption or adopt an incomplete draft with no profile. Recommendation: allow only if correction strategy adoption is otherwise useful and confirmation clearly says the adopted draft remains incomplete.
- One valid profile: replace current stored profile list for the current run with the candidate profile.
- Multiple profiles: preserve the exact profile list if the contract and GUI can represent it. If the current editor can only display one active profile, adoption may preserve all profiles in state but display a summary; unsupported widget representation should be an eligibility warning or block until designed.
- Existing live profile before adoption: replace it after confirmation.
- Candidate partial profile state: adopt only if valid and explicitly accepted as incomplete planning state.

Recommendation: use replace-only profile adoption in the first implementation. No merge.

## OutputPolicy Adoption Rules

OutputPolicy copies into `_guided_draft_output_policy_by_run` only after explicit adoption.

Rules:

- OutputPolicy is revalidated against current source immediately before adoption.
- `output_root=None` may be adopted as incomplete draft state if adoption of incomplete plans is allowed.
- Unsafe output root blocks adoption.
- Output root equal to or inside current completed-run source blocks adoption.
- Output root inside legacy output directories blocks adoption.
- `overwrite=True` blocks adoption initially.
- `separate_from_source_required=False` blocks adoption.
- `legacy_outputs_protected=False` blocks adoption.
- Adoption must not create directories.
- Adoption must not write permission probes.
- Adoption must not test-write.
- Output policy editor widgets synchronize only after successful adoption.
- Export path is not changed by OutputPolicy adoption.

Recommendation: use strict blocking for unsafe roots and unsafe flags in the first implementation.

## Existing Live Draft State Conflict Handling

First adoption implementation should be replace-only, not merge.

Cases:

- Live draft has no strategy/profile/output policy: adoption can proceed without replacement warning, after eligibility passes.
- Live draft has correction choices: confirmation required before replacement.
- Live draft has feature/event profile: confirmation required before replacement.
- Live draft has output policy: confirmation required before replacement.
- Live draft has unsaved widget edits not yet applied: these are not plan state, but confirmation should warn that successful adoption will resync widgets to the adopted live draft state.
- Candidate is partial: confirmation must state the adopted draft will be partial/incomplete.
- Candidate is complete: confirmation still required if live draft state is non-empty.

Non-empty live draft state means any of:

- Stored correction choices exist for the current run.
- Stored feature/event profile exists for the current run.
- Stored output policy exists for the current run.
- Applied draft state exists in the source-scoped stores.

Unsaved widget edits do not count as live draft plan state, but they must be mentioned in the warning because adoption widget sync may replace visible unsaved edits.

Confirmation should summarize:

- Number of existing correction choices that will be replaced.
- Existing feature/event profile presence/count that will be replaced.
- Existing output policy presence that will be replaced.
- Candidate correction choice count.
- Candidate feature/event profile count.
- Candidate output policy state.
- Whether the candidate is partial/incomplete.

If the user cancels, no state mutates.

## Candidate State After Adoption

Options:

- Option A: Keep candidate visible and mark "Adopted into live draft at <time>."
- Option B: Clear candidate after adoption.
- Option C: Keep candidate but separate it from live state.

Recommendation: keep candidate visible and mark it as adopted, while making live draft state the source of truth after adoption.

Rationale:

- The user can compare what was reviewed to what is now live.
- Candidate review remains provenance for the adoption action.
- Clearing the candidate can hide what was just adopted.
- Treating the candidate as source of truth after adoption risks bypassing live draft stores.

After successful adoption:

- Candidate may remain displayed.
- Candidate status should say it has been adopted into live draft.
- Live draft state becomes source of truth for readiness, preview, checklist, and export.
- Candidate should not be used for export or execution.

Failed adoption must not clear candidate unless the source changed or the user opens a different candidate. Candidate remains available for review after a failed adoption attempt.

## Live Readiness, Preview, And Checklist After Adoption

Before adoption:

- Live readiness ignores candidate.
- Live preview ignores candidate.
- Live checklist ignores candidate.
- Candidate review remains separate.

After successful adoption:

- Live readiness refreshes from live draft state.
- Live preview refreshes from live draft state.
- Live checklist refreshes from live draft state.
- Candidate review summary remains separate from live readiness.
- Incomplete adopted plan shows incomplete live readiness.
- Complete adopted plan may show configured planning state, but execution remains blocked until a future Run/RunSpec stage.

Candidate review must never be mixed into live readiness. The transition point is only the successful adoption commit.

## Export Behavior After Adoption

Rules:

- Opening a candidate does not change export path.
- Adoption does not change export path.
- Adoption does not auto-export.
- Export after adoption exports current live draft state.
- Export after adoption must not blindly rewrite or overwrite the imported file.
- Existing export path overwrite protections remain.
- If export path still points to the imported candidate file, export protections should continue to reject overwrite unless the user chooses a new explicit path.

Export remains a separate explicit write action.

## Failure And Rollback Behavior

Adoption must use a validate-then-commit pattern.

Failure cases to handle:

- Candidate disappears or is cleared before adoption.
- Candidate file path no longer exists.
- Candidate no longer matches displayed candidate state.
- Source changes after candidate opened.
- ROI inventory changes or cannot be read.
- Output policy becomes unsafe.
- Validation fails at adoption time.
- Confirmation canceled.
- Unexpected exception while copying state.

Required behavior:

- No partial mutation.
- Compute all replacement state first.
- Revalidate candidate and source before mutation.
- Ask confirmation after replacement state is computed but before mutation.
- Only then mutate live stores.
- If mutation fails, leave previous live draft state intact.
- Candidate remains for review after failed adoption unless source changed.
- Display a clear error.
- No files written.
- No directories created.

Implementation should snapshot previous live draft stores before commit. For a small in-memory mutation, the preferred pattern is to construct new store dictionaries locally, then assign them only after all checks and confirmation pass.

## Internal Implementation Plan For Future Stage

Suggested helpers:

- `_guided_imported_plan_adoption_eligibility(candidate) -> status`
- `_guided_candidate_to_live_draft_state(candidate) -> replacement_state`
- `_apply_guided_imported_plan_to_live_draft(replacement_state)`
- `_confirm_guided_imported_plan_adoption(summary) -> bool`
- `_refresh_guided_after_imported_plan_adoption()`

Suggested commit pattern:

1. Read current candidate from `_guided_imported_plan_candidate`.
2. Verify candidate was opened successfully and still matches `_guided_imported_plan_file_path` and displayed review state.
3. Revalidate schema and contract.
4. Revalidate current source and imported source match.
5. Revalidate current ROI/chunk inventory.
6. Revalidate feature/event profiles.
7. Revalidate OutputPolicy against current source.
8. Compute replacement correction choices, feature/event profiles, and output policy in local variables.
9. Determine whether current live draft state is non-empty.
10. Ask confirmation if needed.
11. Apply all store mutations together.
12. Synchronize feature/event, output policy, and confirm strategy widgets.
13. Refresh live readiness, preview, and checklist.
14. Mark candidate as adopted in the candidate review panel.
15. Do not run anything.
16. Do not write anything.

Replacement state should be a plain in-memory object or dict containing:

- New `_guided_strategy_choices` entries for current run.
- New feature/event profile list for current run.
- New OutputPolicy for current run.
- Adoption warnings to display after commit.
- Candidate identity fields used for audit display.

## Recommended Next Stage

The recommended next stage is 4H2a: adoption eligibility/status only, no mutation.

Reasoning:

- Adoption has enough independent gates that the UI should first expose why a candidate is or is not adoptable.
- Eligibility/status can reuse the read-only candidate review model without mutating live draft state.
- It gives tests a stable target for source, ROI, feature/event, strategy, and OutputPolicy adoption gates before the mutation boundary exists.
- It reduces risk before replace-only adoption is implemented.

After 4H2a is accepted, 4H3 should implement explicit source-matched replace-only adoption, no Run, no RunSpec, and no output writes.
