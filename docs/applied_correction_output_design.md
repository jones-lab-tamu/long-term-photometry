# Applied Correction Output Design

This document defines the planned output contract for future applied correction support. It is design documentation only. It does not implement applied `signal_only_f0`, change dynamic-fit correction, change dF/F calculation, change feature/event detection, or modify HDF5 outputs.

## Core Principle

Applied correction is selected once per ROI recording. The selected strategy is applied consistently to the ROI recording. The pipeline must not create a default mosaic trace that switches correction modes chunk-by-chunk.

Future user-facing choices:

`requested_correction_strategy`

- `dynamic_fit`
- `signal_only_f0`
- `no_correction`
- `auto`

Future applied strategy values:

`applied_correction_strategy`

- `dynamic_fit`
- `signal_only_f0`
- `no_correction`

`auto` is a request/selection mode, not an applied correction strategy. If `requested_correction_strategy = auto`, the recording-level selector chooses one `applied_correction_strategy` for the ROI recording.

## Output Trace Contract

Future trace outputs should remain explicitly identifiable by correction identity.

`dynamic_fit_dff`

dF/F generated by dynamic isosbestic/reference correction.

`signal_only_f0`

The signal-derived F0 baseline trace used for signal-only F0 correction.

`signal_only_f0_dff`

dF/F generated as `(signal - signal_only_f0) / signal_only_f0`.

`applied_dff`

The dF/F trace selected for downstream analysis according to `applied_correction_strategy`.

`applied_signal`

Optional future alias for the signal trace actually used by downstream phasic/event analysis.

Required interpretation constraints:

- `applied_dff` must never be interpreted without `applied_correction_strategy` provenance.
- `dynamic_fit_dff` and `signal_only_f0_dff` should remain separately identifiable.
- `signal_only_f0_dff` must not overwrite or masquerade as `dynamic_fit_dff`.
- If both `dynamic_fit_dff` and `signal_only_f0_dff` are computed, both should be saved or exportable for comparison.

## Suggested Formulas

Dynamic fit:

```text
dynamic_fit_dff = existing dynamic/reference-corrected dF/F output
```

This document does not redefine the existing dynamic/reference correction calculation. See the dynamic fitting and correction documentation/code for current implementation details.

Signal-only F0:

```text
signal_only_f0_dff = (signal - signal_only_f0) / signal_only_f0
```

Where:

```text
signal_only_f0 is the state-aware lower-envelope signal-only F0 candidate.
```

Guardrails:

- `signal_only_f0` must be finite.
- `signal_only_f0` must be positive or above an explicit floor.
- Chunks with insufficient anchors, insufficient low support, large gaps, high extrapolation, high-state/edge-state context, or low confidence must be flagged.
- Guardrail failures should not silently produce trusted `applied_dff`.

## No-Correction Semantics

`no_correction` means no corrected dF/F trace is generated as the applied trace.

`no_correction` must not secretly compute signal-only dF/F. Plotting may still show raw signal, centered signal, or normalized signal, but these should be labeled as uncorrected/visualization traces, not corrected dF/F. Feature detection on `no_correction` outputs requires a separate explicit design decision and should not be assumed.

## Provenance Fields

Future applied outputs must carry explicit strategy and configuration provenance.

Required future fields:

- `requested_correction_strategy`
- `applied_correction_strategy`
- `correction_strategy_selection`
- `correction_analysis_config_hash`
- `correction_analysis_config_summary`
- `correction_analysis_config_path`
- `strategy_evidence_policy`
- `strategy_evidence_grouping_mode`
- `strategy_evidence_generated_at`
- `strategy_evidence_source`
- `applied_trace_source`
- `applied_trace_units`
- `applied_trace_review_required`
- `applied_trace_warning_level`
- `applied_trace_flags`

Allowed `correction_strategy_selection` values:

- `manual`
- `auto`

Allowed `applied_trace_source` values:

- `dynamic_fit_dff`
- `signal_only_f0_dff`
- `none`

Allowed `applied_trace_units` values:

- `dff`
- `raw`
- `normalized`
- `none`

These fields must travel with exported traces, summary CSVs, HDF5 outputs, and feature/event outputs where applicable.

## Feature Detection On Applied Traces

Downstream feature/event detection should run on `applied_dff` only when `applied_correction_strategy` produces a valid `applied_dff`. Feature/event outputs must record which trace was used.

Proposed future fields:

- `feature_detection_input_trace`
- `feature_detection_input_strategy`
- `feature_detection_input_config_hash`
- `feature_detection_review_required`
- `feature_detection_warning_flags`

Allowed `feature_detection_input_trace` values:

- `applied_dff`
- `dynamic_fit_dff`
- `signal_only_f0_dff`
- `raw_signal`
- `none`

Allowed `feature_detection_input_strategy` values:

- `dynamic_fit`
- `signal_only_f0`
- `no_correction`

If `applied_correction_strategy = signal_only_f0`, downstream features must be labeled as derived from `signal_only_f0_dff`. If `applied_correction_strategy = dynamic_fit`, downstream features must be labeled as derived from `dynamic_fit_dff`. If `applied_correction_strategy = no_correction`, feature detection should be disabled or explicitly labeled as uncorrected according to a future design decision.

## Review And Caution Masks

Future masks/annotations:

- `applied_trace_review_mask`
- `applied_trace_caution_mask`
- `applied_trace_review_chunk_ids`
- `applied_trace_caution_chunk_ids`

Meaning:

- `review_mask` marks samples/chunks requiring inspection under the applied global strategy.
- `caution_mask` marks samples/chunks carrying cautionary context.
- These masks do not imply chunkwise switching of correction modes.

For `dynamic_fit`, review/caution masks should be driven by dynamic/reference QC and explicit review flags.

For `signal_only_f0`, review/caution masks should be driven by signal-only F0 QC, anchor support, extrapolation/gap flags, high-state/edge-state context, low confidence, and explicit review flags.

For `no_correction`, review/caution masks explain why correction was not applied and where signal/reference behavior was problematic.

## HDF5/Output Naming Design

Proposed future HDF5/output paths, conceptually:

```text
/roi/<ROI>/correction/dynamic_fit/dff
/roi/<ROI>/correction/dynamic_fit/provenance

/roi/<ROI>/correction/signal_only_f0/f0
/roi/<ROI>/correction/signal_only_f0/dff
/roi/<ROI>/correction/signal_only_f0/provenance

/roi/<ROI>/correction/applied/dff
/roi/<ROI>/correction/applied/provenance
/roi/<ROI>/correction/applied/review_mask
/roi/<ROI>/correction/applied/caution_mask

/roi/<ROI>/features/input_provenance
```

Exact paths may be adjusted during implementation, but the separation between `dynamic_fit`, `signal_only_f0`, and `applied` outputs must be preserved.

## Staged Implementation Plan

1. Design/output contract documentation.
2. Compute and export `signal_only_f0_dff` as a parallel diagnostic trace, not yet applied.
3. Add manual recording-level applied strategy selection: `dynamic_fit`, `signal_only_f0`, `no_correction`.
4. Add `applied_dff` and applied provenance fields.
5. Route feature/event detection to `applied_dff` with provenance.
6. Enable auto to select the applied strategy using recording-level proposals.
7. Add GUI controls after backend provenance is stable.

Auto should not be the first applied-correction implementation step. Manual `signal_only_f0` should be inspectable before auto applies it.

## Open Design Questions

1. Should feature detection run automatically on `signal_only_f0_dff` when `signal_only_f0` is the applied strategy?
2. Should `dynamic_fit_dff` and `signal_only_f0_dff` both be saved for every ROI when computable?
3. What exact visualization trace should be shown for `no_correction`?
4. Should low-confidence `signal_only_f0` be allowed as `applied_dff` with warnings, or blocked in some cases?
5. Should any chunks be automatically excluded from downstream summaries, or only flagged at first?
6. Should `applied_dff` be stored as a separate copied trace, or as a pointer/alias to the selected strategy trace plus provenance?
7. Should auto be allowed to select `no_correction`, or should `no_correction` remain manual-only until better defined?
